# Requirements Document — 精算情報取得

## Introduction

Tateca は旅行・食事・イベントなど日常の割り勘シーンにおける共有費用を管理するアプリケーションである。
精算情報取得は、グループ内の全取引をもとに、メンバー間の最適な精算方法を算出する操作である。

各メンバーの残高（立替の支払者としての貸し / 義務者としての借り）を日本円に換算して算出し、
債務者から債権者への送金を最小化するように精算を最適化する。

為替換算や端数処理による残差が発生する場合は、最大債務者に調整を適用して全メンバーの残高合計がゼロになることを保証する。

## Scope

- グループ内の全取引に基づくメンバー間の残高計算
- 外貨取引の日本円への換算
- 精算の最適化（送金回数の最小化）
- 端数処理による残差の調整

### Out of Scope

- グループメンバーシップの検証（現在の実装では、認証済みであれば任意のグループの精算情報を取得可能）
- グループの存在確認（存在しないグループやメンバーが0人のグループを指定した場合も、空の精算一覧を返却する）
- 日本円以外の通貨での精算金額の表示
- 精算の実行（実際の送金処理）— 精算情報は参考情報であり、返済取引の作成は別操作
- 精算金額の端数の切り上げ・切り捨ての選択（四捨五入で固定）
- 特定のメンバー間の精算のみの算出（グループ全体の精算を算出する）

## Common Preconditions

すべての要件は以下を暗黙の前提とする。

- 要求者は認証済みである

## Domain Definitions

- **精算:** グループ内のメンバー間の貸し借りを解消するために必要な送金の一覧
- **残高:** あるメンバーの貸し借りの差額。正の値は借り（他のメンバーに返済する必要がある金額）、負の値は貸し（他のメンバーから返済を受ける金額）を表す
- **債務者:** 残高が正のメンバー。他のメンバーに対して返済義務がある
- **債権者:** 残高が負のメンバー。他のメンバーから返済を受ける権利がある
- **精算の最適化:** 債務者と債権者を金額順に組み合わせ、送金回数を最小化するアルゴリズム
- **端数調整:** 為替換算の丸めにより残高合計がゼロにならない場合に、最大債務者の残高を調整して合計をゼロにする仕組み
- **精算金額:** 債務者から債権者への送金金額。日本円の整数値（四捨五入）で表される

## Processing Order

1. グループメンバーの取得
2. グループ内の全義務の取得
3. 各メンバーの残高計算（義務金額を為替レートで日本円に換算）
4. 端数調整（残高合計がゼロになるよう最大債務者を調整）
5. 精算の最適化（債務者と債権者のマッチング）
6. 精算一覧の生成

## Requirements

### Requirement 1: 精算情報の算出

**User Story:** 認証済みユーザーとして、グループの精算情報を確認したい。誰が誰にいくら支払えば貸し借りが解消されるかを把握するため。

#### Acceptance Criteria

1. GIVEN 指定されたグループに取引が存在する WHEN 精算情報の取得を要求する THEN システム SHALL 債務者から債権者への精算一覧を日本円で返却する

2. GIVEN 精算情報の取得が完了した WHEN 各精算項目を確認する THEN システム SHALL 送金元（債務者）・送金先（債権者）・送金金額が含まれていること

3. GIVEN 指定されたグループに取引が存在しない WHEN 精算情報の取得を要求する THEN システム SHALL 空の精算一覧を返却する

4. GIVEN 指定されたグループの全メンバーの残高がゼロである WHEN 精算情報の取得を要求する THEN システム SHALL 空の精算一覧を返却する

### Requirement 2: 外貨取引の日本円換算

**User Story:** 認証済みユーザーとして、外貨建ての取引を含むグループの精算情報を確認したい。異なる通貨の取引を統一的に比較するため。

#### Acceptance Criteria

1. GIVEN 外貨建ての取引が存在する WHEN 精算情報の取得を要求する THEN システム SHALL 各取引の通貨に対応する為替レートを使用して日本円に換算し精算金額を算出する

2. GIVEN 複数の通貨の取引が混在する WHEN 精算情報の取得を要求する THEN システム SHALL すべての取引を日本円に換算して精算金額を算出する

### Requirement 3: 端数調整

**User Story:** 認証済みユーザーとして、為替換算の端数処理後も精算金額の整合性が保たれていることを期待する。

#### Acceptance Criteria

1. GIVEN 為替換算の丸めにより残高合計にゼロでない残差が発生する WHEN 精算情報の取得を要求する THEN システム SHALL 最大債務者の残高を調整して全メンバーの残高合計をゼロにする

2. GIVEN 精算情報の取得が完了した WHEN 精算金額を確認する THEN システム SHALL 精算金額が日本円の整数値（四捨五入）で表されていること

### Requirement 4: 精算の最適化

**User Story:** 認証済みユーザーとして、最小限の送金回数で貸し借りを解消したい。精算の手間を減らすため。

#### Acceptance Criteria

1. GIVEN 2人のメンバー間に貸し借りが存在する WHEN 精算情報の取得を要求する THEN システム SHALL 1件の精算（債務者から債権者への送金）を返却する

2. GIVEN 3人以上のメンバー間に循環する貸し借りが存在する WHEN 精算情報の取得を要求する THEN システム SHALL 相殺可能な部分を考慮して精算回数を最小化する

### Requirement 5: 入力バリデーション

**User Story:** システム管理者として、不正な入力による精算情報取得を防ぎたい。

#### Acceptance Criteria

1. WHEN グループ識別子が不正な形式である THEN システム SHALL 入力不備として拒否する
