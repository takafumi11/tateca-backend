# Requirements Document — 取引削除

## Introduction

Tateca は旅行・食事・イベントなど日常の割り勘シーンにおける共有費用を管理するアプリケーションである。
取引削除は、グループ内の取引とそれに紐づく義務をすべて削除する操作である。

立替取引・返済取引のいずれも削除可能であり、取引に紐づく義務が先に削除された後、取引本体が削除される。
削除された取引は復元できない。

## Scope

- 取引とそれに紐づく義務の削除
- 立替取引・返済取引の両方が削除対象

### Out of Scope

- グループメンバーシップの検証（現在の実装では、認証済みであれば任意の取引を削除可能）
- グループ識別子と取引の所属グループの整合性チェック（パスのグループ識別子と取引が実際に所属するグループが異なっていても削除可能）
- 削除の取り消し（Undo）
- 論理削除（ソフトデリート）— 物理削除のみ
- 削除理由の記録
- 削除の通知（プッシュ通知など）
- 削除時の精算情報の再計算

## Common Preconditions

すべての要件は以下を暗黙の前提とする。

- 要求者は認証済みである

## Domain Definitions

- **取引削除:** 取引とそれに紐づくすべての義務をシステムから完全に削除する操作
- **義務:** 立替取引における個々の負担、または返済取引における返済記録

## Processing Order

1. 取引に紐づく義務の削除
2. 取引本体の削除

## Requirements

### Requirement 1: 取引の削除

**User Story:** 認証済みユーザーとして、誤って作成した取引を削除したい。正確な費用記録を維持するため。

#### Acceptance Criteria

1. GIVEN 指定された取引が存在する WHEN 取引削除を要求する THEN システム SHALL 取引とそれに紐づくすべての義務を削除する

2. GIVEN 取引削除が完了した WHEN グループの取引履歴を確認する THEN システム SHALL 削除された取引が含まれていないこと

### Requirement 2: 入力バリデーション

**User Story:** システム管理者として、不正な入力による取引削除を防ぎたい。

#### Acceptance Criteria

1. WHEN グループ識別子が不正な形式である THEN システム SHALL 入力不備として拒否する

2. WHEN 取引識別子が不正な形式である THEN システム SHALL 入力不備として拒否する

### Requirement 3: リソース不存在

**User Story:** 要求者として、存在しない取引の削除には適切なエラーを受け取りたい。問題の原因を理解するため。

#### Acceptance Criteria

1. GIVEN 認証済みユーザー WHEN 存在しない取引の削除を要求する THEN システム SHALL リソース不在として拒否する
