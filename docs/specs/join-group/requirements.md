# Requirements Document — グループ参加

## Introduction

Tateca は旅行・食事・イベントなど日常の割り勘シーンにおける共有費用を管理するアプリケーションである。
グループ参加は、グループ作成時に名前のみで登録された未参加メンバーが、招待トークンを使用してグループに正式に参加する操作である。

参加により、未参加メンバーと認証済みアカウントが紐付けられ、そのメンバーとしてグループ内で操作（取引の記録等）ができるようになる。

## Scope

- 招待トークンによるグループへの参加
- 既存メンバーシップの重複チェック
- グループ参加上限のチェック
- 招待トークンの検証

### Out of Scope

- 招待トークンの有効期限チェック（現在の実装ではトークンの一致のみを検証。`tokenExpires` フィールドは存在するが参照されていない）
- 指定メンバーが当該グループに所属していることの検証（現在の実装ではメンバーの存在のみ確認し、当該グループに所属しているかは未検証）
- 招待トークンの再発行
- 参加の通知（プッシュ通知など）
- 参加の取り消し（Undo）— 脱退（leave-group）で対応

## Common Preconditions

すべての要件は以下を暗黙の前提とする。

- 要求者は認証済みである
- 要求者のアカウント情報が存在する

## Domain Definitions

- **招待トークン:** グループ作成時に自動生成される一意のトークン。グループへの参加権限を証明するために使用される
- **未参加メンバー:** 認証済みアカウントと紐付いていないメンバー。名前のみで登録された状態にある
- **グループ参加:** 未参加メンバーに認証済みアカウントを紐付け、そのメンバーとしてグループに参加する操作
- **グループ参加上限:** 1人の認証済みユーザーが参加できるグループ数の上限（9グループ）。特権ユーザーはこの制限を免除される
- **特権ユーザー:** グループ参加上限の制限を免除される特定の認証済みユーザー。システム設定により指定される

## Processing Order

1. グループの存在確認
2. 既存メンバーシップの重複チェック（同一認証アカウントが既にグループに参加しているか）
3. グループ参加上限のチェック
4. 招待トークンの検証
5. メンバーと認証済みアカウントの紐付け

## Requirements

### Requirement 1: グループへの参加（正常系）

**User Story:** 認証済みユーザーとして、招待トークンを使用してグループに参加したい。グループ内の共有費用管理に参加するため。

#### Acceptance Criteria

1. GIVEN 要求者が認証済みであり、指定されたグループが存在し、招待トークンが正しく、要求者がグループ参加上限に達していない WHEN グループ参加を要求する THEN システム SHALL 指定されたメンバーに要求者の認証済みアカウントを紐付ける

2. GIVEN グループ参加が完了した WHEN グループ情報を確認する THEN システム SHALL 参加したメンバーが認証済みメンバーとして含まれること

### Requirement 2: 重複参加の防止

**User Story:** システム管理者として、同一ユーザーがグループに重複して参加することを防ぎたい。データの整合性を保つため。

#### Acceptance Criteria

1. GIVEN 要求者が既にグループの認証済みメンバーである WHEN 同じグループへの参加を要求する THEN システム SHALL ビジネスルール違反として拒否する

2. GIVEN 要求者の認証済みアカウントが別のメンバーとして既にグループに紐付いている WHEN 別のメンバーとしてグループ参加を要求する THEN システム SHALL ビジネスルール違反として拒否する

### Requirement 3: グループ参加上限

**User Story:** システム管理者として、1人のユーザーが過度に多くのグループに参加することを制限したい。

#### Acceptance Criteria

1. GIVEN 要求者が既に9グループに参加している WHEN 新しいグループへの参加を要求する THEN システム SHALL ビジネスルール違反として拒否する

2. GIVEN 要求者が特権ユーザーである WHEN グループ参加上限を超えた状態でグループ参加を要求する THEN システム SHALL グループ参加上限を免除し、参加を許可する

### Requirement 4: 招待トークンの検証

**User Story:** システム管理者として、不正なトークンによるグループ参加を防ぎたい。グループのセキュリティを保つため。

#### Acceptance Criteria

1. GIVEN 要求者が認証済みである WHEN 不正な招待トークンでグループ参加を要求する THEN システム SHALL 権限不足として拒否する

### Requirement 5: 入力バリデーション

**User Story:** システム管理者として、不正な入力によるグループ参加を防ぎたい。

#### Acceptance Criteria

1. WHEN メンバー識別子が未指定である THEN システム SHALL 入力不備として拒否する

2. WHEN 招待トークンが未指定である THEN システム SHALL 入力不備として拒否する

3. WHEN メンバー識別子が不正な形式である THEN システム SHALL 入力不備として拒否する

4. WHEN グループ識別子が不正な形式である THEN システム SHALL 入力不備として拒否する

### Requirement 6: リソース不存在

**User Story:** 要求者として、存在しないリソースに対する操作には適切なエラーを受け取りたい。問題の原因を理解するため。

#### Acceptance Criteria

1. GIVEN 認証済みユーザー WHEN 存在しないグループへの参加を要求する THEN システム SHALL リソース不在として拒否する

2. GIVEN 認証済みユーザー WHEN 存在しないメンバーを指定してグループ参加を要求する THEN システム SHALL リソース不在として拒否する

3. GIVEN 認証済みだがアカウント情報が存在しない WHEN グループ参加を要求する THEN システム SHALL リソース不在として拒否する
